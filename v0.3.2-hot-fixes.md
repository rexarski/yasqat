# § A. Fundamental — sequence type system

## A1. Merge StateSequence and EventSequence?

**Finding.** Both classes live in `src/yasqat/core/sequence.py` and are
structurally identical:

- Same schema `[id_column, time_column, state_column]` (long format).
- Same `__init__` signature (`sequence.py:108-115` vs `:305-311`).
- Same `_validate_and_prepare` and `_infer_alphabet` implementations
  (`:117-130` mirrors `:313-326`; `:132-134` mirrors `:328-330`).
- Same inherited `data`, `alphabet`, `config`, `n_sequences`, `sequence_ids`.
- `StateSequence.to_event_sequence()` (`:239`) and
  `EventSequence.to_state_sequence()` (`:361`) are both no-op reshapes — they
  just re-wrap the same DataFrame under a different class name.

Unique methods:

- StateSequence: `to_sts`, `to_sps`, `to_dss`, `to_event_sequence`,
  `to_interval_sequence`, `encode_states`, `get_states_for_sequence`.
- EventSequence: only `event_counts()` (a trivial group-by count) and
  `events_per_sequence()` (another trivial group-by count).

**Verdict.** Design-gap. The two types carry almost no semantic distinction in
the code. The intended difference — "state = temporal extent, event =
instantaneous occurrence" — is not reflected in data shape, validation, or any
algorithm that consumes the pool.

**Recommendation.** Drop `EventSequence` in favour of a unified `StateSequence`
with two tiny helpers:

```python
# on StateSequence
def state_counts(self) -> pl.DataFrame: ...        # was event_counts
def observations_per_sequence(self) -> pl.DataFrame: ...  # was events_per_sequence
```

Users who want "event-like" semantics can read a long-format DataFrame into
`StateSequence` as they do today. Intervals remain distinct because they carry
real extra structure (start/end columns, duration, overlaps).

**Why this is safer than it looks.** `SequencePool.compute_distances` and every
metric/statistic function operates on `StateSequence` / `SequencePool`, never on
`EventSequence` directly. Removing `EventSequence` has zero callers in
`src/yasqat/metrics`, `src/yasqat/statistics`, or `src/yasqat/clustering` — only
`tests/` and the class's own `to_state_sequence()` method need to change.

## A2. Contradicting `config` and `alphabet` silently accepted

**Finding.** `StateSequence.__init__` (`sequence.py:108-115`) stores whichever
`config` and `alphabet` the user passes, without comparing them. If the user
supplies an alphabet built from a different column (or from a disjoint set of
states), the mismatch is silent until an encode/decode misaligns downstream.

**Verdict.** Bug — validation hole at a public API boundary.

**Recommendation.** Add a one-line check at the end of `__init__` (applies to
`StateSequence`, `EventSequence` if kept, `IntervalSequence`, and
`SequencePool`):

```python
observed = set(self._data[self._config.state_column].unique().to_list())
declared = set(self._alphabet.states)
missing = observed - declared
if missing:
    raise ValueError(
        f"Alphabet does not cover all observed states. Missing: {sorted(missing)}. "
        f"Alphabet has {len(declared)} states; data contains {len(observed)}."
    )
```

The reverse (alphabet contains extra states not in data) is fine — users often
declare a domain alphabet larger than what happens to appear in a sample.

## A3. `to_event_sequence` / `to_interval_sequence` — why do they exist?

**Finding.**

- `StateSequence.to_event_sequence()` (`sequence.py:239`): identity reshape —
  returns the same data wrapped as `EventSequence`. Only callers are in
  `tests/`.
- `StateSequence.to_interval_sequence()` (`sequence.py:258`): real work — runs
  `to_sps()` and re-labels `start` → `start_column`, `end` → `end_column`. This
  is the "collapse a tick-by-tick sequence into spells" bridge, and it's
  genuinely useful for interval-based analyses. Only test callers in `src/`.
- `IntervalSequence.to_event_sequence()` (`sequence.py:591`): sensible — emits
  one event per interval *start*. Useful for event-count analyses that ignore
  duration.

**Verdict.** `to_event_sequence` is vestigial; `to_interval_sequence` is
meaningful but undocumented.

**Recommendation.**

- If A1 is accepted (merge state/event): drop both `to_event_sequence`
  overloads.
- Keep `StateSequence.to_interval_sequence` and document it as "collapse
  consecutive same-state ticks into intervals (run-length encoding)".

## A4. Event→State loses `end_column`?

**Finding.** `EventSequence.to_state_sequence()` at `sequence.py:361` selects
only `[id_column, time_column, state_column]`. There is no `end_column` to
lose — `EventSequence` never stored one; only `IntervalSequence` has start/end.

**Verdict.** Non-bug — result of a misconception about what EventSequence holds.

**Recommendation.** No code change. If A1 is rejected, add a docstring on
`EventSequence` clarifying that events are instantaneous (no duration) by
design — round-tripping `IntervalSequence → EventSequence → StateSequence` does
lose durations, but that's because the event abstraction discards them
intentionally.

## A5. Utility of `to_sts`, `to_sps`, `to_dss`

- `to_sps()` — called by `statistics/descriptive.py:48`,
  `visualization/timeline.py:89`, and indirectly through

- `to_sts()` — zero callers in `src/`. Essentially `self._data` unchanged.

**Verdict.** `to_sps` keep. `to_sts`/`to_dss` are TraMineR-parity sugar with no

I'd lean toward option 2: drop `to_sts` entirely (it's `self.data` with a
different name, which is noise), keep `to_dss`.

## A9. `IntervalSequence.to_state_sequence(time_points)` too slow

**Finding.** `sequence.py:525-589`. Algorithm: build a `time_points` DataFrame,
then loop serially over sequences; for each, do a cross-join of
`time_points × intervals`, filter by `time >= start AND time < end`, group by
time, take first state.

Complexity: **O(n_sequences × n_intervals_per_seq × n_time_points)**. For
100 sequences × 10 intervals × 10⁶ time points that's 10⁹ joined rows before
filtering.

What it does semantically: samples a stepwise piecewise-constant state
function at each requested time.

**Verdict.** Performance bug.

**Recommendation.** Replace the per-sequence loop with a single vectorized
`join_asof` (polars):

```python
# Per sequence-group, join each time point to the interval whose
# [start, end) contains it. join_asof with strategy='backward' on start_col
# picks the most recent interval at or before t; then filter t < end.
tp = pl.DataFrame({"t": time_points}).with_columns(pl.col("t").cast(pl.Int64))
out = (
    self._data.sort([id_col, start_col])
    .join_asof(
        tp, left_on=start_col, right_on="t", by=id_col, strategy="backward"
    )
    .filter(pl.col("t") < pl.col(end_col))
    .select(id_col, pl.col("t").alias(time_col), state_col)
)
```

Drops complexity to roughly **O((n_intervals + n_time_points) · log)**.

## B2. Optimal matching `(35,35) → (33,33)` broadcast error

**Finding.** `src/yasqat/metrics/optimal_matching.py:101-121`. The recent fix
(commit 89577ca) added this validation:

```python
if sm_matrix.shape[0] <= max_state or sm_matrix.shape[1] <= max_state:
    raise ValueError(...)
```

This only rejects **undersized** matrices. Your error case — a `(35,35)` matrix
used against a pool whose sequences cap at state index 32, expecting a
`(33,33)` matrix — passes the guard (35 > 32) but then fails later with the
numpy broadcast error.

Root cause: your `sub_cost` was built from a *larger* alphabet than the pool
actually encodes. Most likely cause: you called
`substitution_cost_matrix(pool_big)` and then `optimal_matching_distance` on
sequences encoded via `pool_small.get_encoded_sequence(...)`.

**Verdict.** Bug — validation is too loose. Letting an oversized matrix through
is not safer than rejecting it; the "help" gets deferred to a cryptic numpy
error.

**Recommendation.** Tighten the guard to require an exact match:

```python
n_states = seq_pool_size  # needs to be passed in or inferred from alphabet
if sm_matrix.shape != (n_states, n_states):
    raise ValueError(
        f"Substitution matrix shape {sm_matrix.shape} does not match alphabet "
        f"size ({n_states}, {n_states}). Build it from the same pool that "
        f"encoded the sequences: `substitution_cost_matrix(pool)`."
    )
```

If the low-level `optimal_matching_distance(seq_a, seq_b, ...)` doesn't have
access to an alphabet, use `max(seq_a.max(), seq_b.max()) + 1` as the required
size — but still reject oversized matrices with the same clear error.

## B3. `modal_states(granularity=…)` should accept time-unit strings

**Finding.** `statistics/descriptive.py:635-666`. `granularity: int | None` is
an integer bucket size: `granularity=7` means "group into 7-unit buckets". It
is time-unit-agnostic because yasqat time columns are just numeric — no
underlying "day" vs "hour" metadata.

**Verdict.** Design choice, not a bug. But the user's expectation is reasonable.

**Recommendation.** Add a paired `time_unit` parameter for ergonomics:

```python
def modal_states(
    sequence, *,
    granularity: int | str | None = None,
    time_unit: Literal["second","minute","hour","day","week","month","quarter","year"] | None = None,
) -> pl.DataFrame:
    if isinstance(granularity, str):
        if time_unit is None:
            raise ValueError(
                "When granularity is a time-unit string, time_unit must specify "
                "the resolution of the time column (e.g. time_unit='day')."
            )
        granularity = _resolve_granularity(granularity, time_unit)
    ...
```

`_resolve_granularity("week", "day")` → `7`. `_resolve_granularity("month",
"day")` → `30` (calendar-approximation caveat in docstring).

This preserves the current integer API and adds the string ergonomics the user
wants without guessing units.

# § C. Statistics

## C1. `discrepancy_analysis` returns `p=0.0010` (or N/A) under perfect separation

**Finding.** `statistics/discrepancy.py`. When observed
`pseudo_F = ∞` (perfect group separation, `within_ss = 0`), the permutation
loop at `:120-129` does `if perm_f >= pseudo_f:` which evaluates
`finite >= inf` as `False` for every permutation. So `n_extreme = 0` and
`p_value = 1/(n_permutations + 1) = 0.0010` for 999 permutations.

The `p=N/A` variant appears when `n_permutations = 0` or the observed F was
already `inf` and some earlier branch short-circuited.

Real data triggers this more often than synthetic because real trajectories
include duplicates within groups (identical sequences), which zero out
within-group distance sums.

**Verdict.** Bug.

**Recommendation.** Special-case infinite observed F:

```python
n_extreme = 0
if np.isinf(pseudo_f):
    for _ in range(n_permutations):
        perm_labels = rng.permutation(labels)
        perm_within = _compute_ss(dist_matrix, perm_labels)[1]
        perm_between = perm_total - perm_within
        perm_f = _compute_pseudo_f(perm_between, perm_within, n, perm_labels)
        if np.isinf(perm_f):   # count infinite permutations only
            n_extreme += 1
else:
    for _ in range(n_permutations):
        ...
        if perm_f >= pseudo_f:
            n_extreme += 1
p_value = (n_extreme + 1) / (n_permutations + 1)
```

Add a warning when `pseudo_f = inf` so the result carries a hint.

## C2. `dissimilarity_tree()` `UFuncTypeError` on categorical covariates

**Finding.** `statistics/disstree.py:101-103` inside `_find_best_split`:

```python
for split_val in unique_vals[:-1]:
    left_mask = values <= split_val   # fails on string dtype
```

The code assumes numeric covariates. Passing an object-dtype / string-dtype
column (`"urban"`/`"rural"`, job categories, education levels) triggers
`UFuncTypeError: ufunc 'less_equal' did not contain a loop ...`.

**Verdict.** Bug — common user workflow not supported, error is cryptic.

**Recommendation.** Branch on dtype in `_find_best_split`:

```python
is_numeric = np.issubdtype(values.dtype, np.number)
for split_val in unique_vals[:-1]:
    if is_numeric:
        left_mask = values <= split_val
    else:
        left_mask = values == split_val   # one-vs-rest for categoricals
    ...
```

And in `_build` mirror the same branch when recreating the mask at the split.
Store `split_value` as `float | str` in `DissTreeNode`; update the
pretty-printer accordingly.

## D2. `pam_range(dm, k_range(2, 10))` only returns k=2 and k=10

**Finding.** `clustering/quality.py:196-260`. The parameter name is
`k_range`, and the loop at `:244` is `for k in k_range:`. If a user passes a
tuple `(2, 10)` they iterate exactly the two endpoints.

There is **no** `k_range` helper function in the codebase — the user's call
`pam_range(dm, k_range(2, 10))` either hits a `NameError` or (if they had
an accidentally-defined `k_range(a, b) -> tuple`) returns only the endpoints.

**Verdict.** UX trap — not a code bug, but easy to misuse.

**Recommendation.** Two small additions:

1. Add a helper for the obvious ergonomic call:

   ```python
   def k_range(start: int, end: int) -> range:
       """Inclusive range of k values: k_range(2, 10) → range(2, 11)."""
       return range(start, end + 1)
   ```

2. Auto-expand 2-tuples in `pam_range` with a warning:

   ```python
   if isinstance(k_range, tuple) and len(k_range) == 2:
       warnings.warn(
           "pam_range received a 2-tuple; treating as (start, end) inclusive. "
           "Pass `range(start, end+1)` or `k_range(start, end)` explicitly."
       )
       k_range = range(k_range[0], k_range[1] + 1)
   ```

Also rename the parameter to `k_values` in the docstring (internal alias kept
for backwards compatibility).

# § E. Visualization

## E1. Index plot `sort_by` accepts nonsense values

**Finding.** `visualization/index_plot.py:132-170`. `sort_by` is typed
`Literal["from.start","from.end","length",None]` but runtime validation is
absent — the helper has an `if/elif` chain with no final `else`, so any
unrecognized string falls through and returns data unsorted.

**Why real-data plots look segmented but synthetic ones don't.** Real data
arrives in arbitrary insertion order, so unsorted rendering produces chaos.
Synthetic generators emit rows in natural sequence order, so "unsorted" happens
to coincide with "sensible order". The semantic options — `"from.start"`,
`"from.end"`, `"length"` — all rank sequences against the time axis
(`config.time_column`), so they are sensitive to missing/irregular timestamps
too.

On `start_ts`: it is **not** a yasqat-recognized column. If your DataFrame
uses `start_ts`, you must configure
`SequenceConfig(time_column="start_ts", ...)` — otherwise the plot looks at
whatever `config.time_column` points to (default `"time"`).

**Verdict.** Bug — silent fallthrough masking a configuration mistake.

**Recommendation.** Add a runtime guard:

```python
_VALID_SORTS = {"from.start", "from.end", "length", None}
if sort_by not in _VALID_SORTS and sort_by not in data.columns:
    raise ValueError(
```

# § F. Filters

## F1. `PatternCriterion` performance

**Finding.** `filters/criteria.py:263-303`. The current loop:

1. Iterates `seq_id in sequence.sequence_ids` in pure Python.
2. Filters and sorts `sequence.data` per sequence (redundant).
3. Recompiles the same regex every iteration (line 277 is inside the loop).
4. Joins states in Python and `re.fullmatch` per sequence.

Complexity: **O(n_sequences × mean_seq_length)** with constant-factor
overhead from per-sequence polars filtering.

**Verdict.** Perf bug (all other criteria are polars-vectorized).

**Recommendation.** Two stages:

1. **Trivial win**: hoist the regex compile out of the loop.
2. **Vectorize** via polars `str.concat` + `str.contains`:

   ```python
   sep = "\x00"
   sorted_ = sequence.data.sort([id_col, time_col])
   grouped = sorted_.group_by(id_col, maintain_order=True).agg(
       pl.col(state_col).str.concat(sep).alias("_joined")
   )
   pattern = self._simple_to_regex(...)  # same helper
   matched = grouped.filter(pl.col("_joined").str.contains(pattern))
   return matched[id_col].to_list()
   ```

That's a single polars expression; no Python loop, no per-sequence filter.

## F2. `PatternCriterion "state_A-?-state_B"` returns empty

**Finding.** `filters/criteria.py:305-358` in `_simple_to_regex`. Given
`"state_A-?-state_B"`, the method splits on `-` into
`["state_A", "?", "state_B"]` and emits regex parts
`["state_A", "(?:[^\x00]+)?", "state_B"]`. The final step **joins these with
the null-byte separator**:

```python
return escaped_sep.join(regex_parts)
```

producing `state_A\x00(?:[^\x00]+)?\x00state_B`. The two literal `\x00`
separators are always required, but the wildcard group (which sits between
them) is optional. For a sequence `A→B` (no middle), the "joined" string is
`state_A\x00state_B` — only one separator, so the regex can never match.

**Verdict.** Bug. Same structural issue applies to `*` and `+` wildcards if
users hit them with zero/one-middle cases.

**Recommendation.** Fold the separator inside the wildcard groups so the
whole "middle slot" is optional together:

```python
regex_parts = []
for i, part in enumerate(parts):
    if part == "*":
        regex_parts.append(f"(?:{escaped_sep}{not_sep}+)*")
    elif part == "+":
        regex_parts.append(f"(?:{escaped_sep}{not_sep}+)+")
    elif part == "?":
        regex_parts.append(f"(?:{escaped_sep}{not_sep}+)?")
    elif part:
        if i == 0:
            regex_parts.append(re.escape(part))
        else:
            regex_parts.append(escaped_sep + re.escape(part))
return "".join(regex_parts) if regex_parts else ""
```

Now `"A-?-C"` compiles to `A(?:\x00[^\x00]+)?\x00C` which matches both
`A\x00B\x00C` (one middle) and `A\x00C` (no middle) — as expected.

Add tests for all three wildcards in the presence of zero/one/two middle
states, since the same mis-join affected all of them.
