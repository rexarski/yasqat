# yasqat v0.4.0 — design

**Status:** approved (brainstorm complete 2026-05-11)
**Type:** breaking release
**Source items:** 5 user-supplied changes + 1 deferred OM bug
**Branch:** `dev`
**Target version:** `0.3.2` → `0.4.0`

## 1. Theme & scope

**Theme:** *less restraint on the consumer, more robust core.* Every change is
either a deletion or a consolidation. The release adds three things only:
two new analytical methods on `StateSequence` (`state_counts`,
`state_per_sequence`) and one classmethod constructor
(`StateSequence.from_intervals`).

### In scope

1. **Hard-unify the sequence model.** `StateSequence` becomes the only
   sequence class. `IntervalSequence` and `BaseSequence` are deleted.
2. **Extend `StateSequence` method surface.** Add `state_counts()`,
   `state_per_sequence()`, `duration()`, `total_duration_by_state()`,
   `spells_per_sequence()`, `span()`. Drop `overlapping_intervals()` and
   `has_overlaps()` (impossible by construction under the unified model).
3. **Add `StateSequence.from_intervals(...)`** as a classmethod that lifts
   the existing interval-to-state-grid sampling logic out of the deleted
   `IntervalSequence`. Delete `infer_sequence_type()`. Loaders return only
   `StateSequence`.
4. **Delete the `visualization` module.** Remove `src/yasqat/visualization/`,
   `tests/test_visualization/`, `demo_showcase.ipynb`, and the
   `plotnine`/`matplotlib` dependencies.
5. **Delete `PatternCriterion`.** Document `QueryCriterion` (or a direct
   polars expression) as the replacement.

### Out of scope

- **OM substitution-matrix bug.** Deferred — see
  `2026-05-11-om-subcost-investigation.md`. No repro yet; will be picked up
  after v0.4.0 ships or sooner if a sharp repro arrives.
- **CI Actions bump** (`actions/checkout@v4` → `@v5`,
  `astral-sh/setup-uv@v5` → `@v6`). Already landed in the same session as
  this design but committed separately.
- **No opportunistic refactors.** Anything not on the user's list stays as
  it is.

### Non-goals

- **No deprecation cycle.** Pre-1.0 package, clean cut. Removed names get a
  CHANGELOG note but no `DeprecationWarning` shim.
- **No replacement visualization library or "starter recipe" docs.** Users
  get full liberty over how they plot the polars DataFrames yasqat returns.

## 2. The unified `StateSequence`

### 2.1 Class shape

`BaseSequence` ABC and `IntervalSequence` are deleted. `StateSequence`
becomes a concrete class with no inheritance hierarchy and no abstract
methods. The constructor signature is unchanged:

```python
StateSequence(data: pl.DataFrame, config: SequenceConfig | None = None,
              alphabet: Alphabet | None = None)
```

The existing `_validate_and_prepare`, `_infer_alphabet`,
`_validate_alphabet_covers_data`, and `_apply_granularity` helpers stay,
but as plain methods rather than abstract overrides.

`SequenceConfig` keeps `id_column`, `time_column`, `state_column`,
`granularity`. The `start_column` and `end_column` fields are deleted —
they only existed for `IntervalSequence`.

### 2.2 Method surface

**Existing methods, kept unchanged:** `data`, `alphabet`, `config`,
`__len__`, `n_sequences`, `sequence_ids`, `get_sequence`, `to_sts`,
`to_sps`, `to_dss`, `encode_states`, `get_states_for_sequence`.

**Removed methods:** `overlapping_intervals`, `has_overlaps`,
`to_state_sequence` (the last one migrates into `from_intervals`).

**New methods** (all return `pl.DataFrame`):

| Method | Signature | Returns |
|---|---|---|
| `state_counts()` | `() -> pl.DataFrame` | `state, count` — pool-wide row count per state. |
| `state_per_sequence(*, proportion: bool = False)` | counts default; pass `proportion=True` to normalize within sequence. | `id, state, count` (or `id, state, proportion`). |
| `duration()` | `() -> pl.DataFrame` | Thin alias: `self.to_sps().select([id_col, state_col, "duration"])`. |
| `total_duration_by_state()` | `() -> pl.DataFrame` | `state, total_duration` aggregated from spell durations. |
| `spells_per_sequence()` | `() -> pl.DataFrame` | `id, n_spells` (renamed from the old `intervals_per_sequence`). |
| `span()` | `() -> pl.DataFrame` | `id, first, last, span`. Works for integer and datetime time columns; for datetime, `span` is a `pl.Duration`. |

### 2.3 The `from_intervals` constructor

```python
@classmethod
def from_intervals(
    cls,
    data: pl.DataFrame,
    *,
    time_points: list[int] | None = None,
    config: SequenceConfig | None = None,
    alphabet: Alphabet | None = None,
    id_col: str = "id",
    start_col: str = "start",
    end_col: str = "end",
    state_col: str = "state",
) -> StateSequence:
    ...
```

Implementation: lift the body of the current
`IntervalSequence.to_state_sequence` (sequence.py:476–579) into this
classmethod, then construct a `StateSequence` from the sampled grid.

**Semantics preserved:** "latest start tiebreak" from v0.3.2 (when multiple
intervals contain the same time point, the one with the largest `start`
wins). Backed by `polars.join_asof(strategy="backward")` exactly as today.

**Datetime handling:** when `time_points=None` and the start/end columns
are datetime, raise a clear `ValueError` directing the user to pass
`time_points` explicitly. The current implicit `int(float(...))` cast on
datetime would crash; we make the failure intentional and informative.
There is no auto-generated daily/hourly grid — the "right" granularity is
domain-specific and a silent default would surprise users.

### 2.4 What this lets us delete

- `BaseSequence` ABC (sequence.py:44–172)
- `IntervalSequence` class (sequence.py:326–601)
- `start_column`, `end_column` fields from `SequenceConfig`
- `infer_sequence_type` from `io/loaders.py`
- All `StateSequence | IntervalSequence` union types throughout
  `io/loaders.py`
- `tests/test_io/test_loaders.py` interval test cases and
  `infer_sequence_type` tests
- `tests/test_core/test_sequence.py::TestIntervalSequence`

## 3. IO module + removals

### 3.1 `io/loaders.py` simplification

All loaders narrow their return types from `StateSequence | IntervalSequence`
to `StateSequence`:

```python
def load_csv(...) -> StateSequence: ...
def load_parquet(...) -> StateSequence: ...
def load_json(...) -> StateSequence: ...
def load_dataframe(...) -> StateSequence: ...
```

The `sequence_type` parameter on `load_dataframe` is **dropped outright** —
no warning shim, no transition period. Users with interval-shaped input
read their own CSV/parquet with polars and call
`StateSequence.from_intervals(df, time_points=...)`. The loaders are
deliberately state-only.

`io/__init__.py` exports shrink to: `load_csv`, `load_parquet`,
`load_json`, `load_dataframe`. `infer_sequence_type` is removed from
`__all__`.

### 3.2 Visualization removal (atomic delete)

- `src/yasqat/visualization/` — entire directory (7 modules, 1296 LOC).
- `tests/test_visualization/` — entire directory.
- `demo_showcase.ipynb` — deleted, not rewritten.
- Top-level imports — there are no `yasqat.visualization` re-exports in
  `src/yasqat/__init__.py`, confirmed by grep. Just dropping the subpackage
  is enough.
- `pyproject.toml` — drop `plotnine` and `matplotlib` from
  `[project] dependencies`. Verify matplotlib is not transitively needed by
  any retained dependency before removal (`numba`, `polars`, `numpy`,
  `scipy` do not pull it).
- `Alphabet.colors` — **kept**. Pure data, costs nothing to retain, and is
  exactly the kind of "use it however you want" hand-off the new philosophy
  implies.

### 3.3 PatternCriterion removal

- Delete `PatternCriterion` class from `filters/criteria.py` (~145 LOC
  including the dual-separator `_simple_to_regex` helper).
- Drop from `filters/__init__.py` `__all__` and re-exports.
- Delete `TestPatternCriterion` from `tests/test_filters/test_criteria.py`.
- Add CHANGELOG migration note with a polars-direct example (see §4.2).

### 3.4 Net dependency change

- **Removed:** `plotnine`, `matplotlib` (verify), and any `viz`/`docs`
  extras that referenced them.
- **Kept:** `polars`, `numpy`, `numba`, `scipy`.
- **Considered & rejected:** keeping plotnine as an optional `[viz]` extra.
  Drops it entirely — matches the "less restraint" theme.

The transitive dep tree under `plotnine` is large (`mizani`, `patsy`,
`pandas`, `pillow`, ...). Worth measuring install-size delta for a
CHANGELOG mention if the drop is meaningful.

## 4. Versioning, migration, test plan

### 4.1 Version & branching

- **Version:** `0.3.2` → `0.4.0` (minor bump signals breaking change in
  pre-1.0 semver).
- **Branch:** feature work on `dev`. Use the `yasqat-release` skill when
  ready to ship — matches the v0.3.2 release pattern (PR `dev` → `main`,
  tag `v0.4.0`, OIDC publish, GH Release with assets).
- **CHANGELOG:** new top section `## [0.4.0] — YYYY-MM-DD` with four
  subsections:
  - `### Removed` (the longest — `IntervalSequence`,
    `infer_sequence_type`, `visualization` module, `PatternCriterion`,
    `plotnine`/`matplotlib` deps, `start_column`/`end_column`,
    `intervals_per_sequence` name)
  - `### Added` (`StateSequence.from_intervals`, six new `StateSequence`
    methods)
  - `### Changed` (loader return types narrowed to `StateSequence`)
  - `### Migration` (the three recipes below)

### 4.2 Migration recipes (CHANGELOG body)

Three short recipes:

1. **Interval-shaped data** — replace `IntervalSequence(df)` with
   `StateSequence.from_intervals(df, time_points=[...])`.

2. **Pattern matching** — replace `PatternCriterion("A-*-C")` with a polars
   expression directly:

   ```python
   matched_ids = (
       seq.data
       .group_by("id")
       .agg(pl.col("state").str.join("-").alias("s"))
       .filter(pl.col("s").str.contains(r"^A-[^-]+-C$"))
       .get_column("id")
       .to_list()
   )
   ```

   Be upfront: this is more code than `PatternCriterion("A-*-C")` was. The
   tradeoff is "verbose but transparent" instead of "concise but fragile".

3. **Visualization** — pull `to_sps()` / `state_distribution()` / etc. into
   your tool of choice (matplotlib, altair, observable). No replacement
   library is provided.

### 4.3 Test plan

**Deletions** (mechanical):
- `tests/test_visualization/` — entire directory.
- `tests/test_core/test_sequence.py::TestIntervalSequence` — class.
- `tests/test_io/test_loaders.py` — `infer_sequence_type` tests, interval
  test cases.
- `tests/test_filters/test_criteria.py::TestPatternCriterion` — class.

**New tests** in `tests/test_core/test_sequence.py`:
- `TestStateSequenceMethods` — one test per new method:
  `state_counts`, `state_per_sequence` (counts + proportion modes),
  `duration`, `total_duration_by_state`, `spells_per_sequence`,
  `span` (integer + datetime).
- `TestFromIntervals` — happy path, datetime + `time_points=None` raises,
  datetime + explicit `time_points` works, alphabet mismatch raises,
  latest-start tiebreak preserved.

**Migrated tests:** existing `IntervalSequence(...).to_state_sequence(...)`
behavioural tests → `StateSequence.from_intervals(...)` with the same
inputs and expected outputs. Guards against regression in the lifted code
path.

**Coverage target:** keep current line coverage on what remains; do not
chase coverage on deleted modules.

### 4.4 Order of work

The order minimizes broken intermediate states:

1. Add new `StateSequence` methods + `from_intervals` classmethod
   (additive, tests first per TDD).
2. Migrate any internal `IntervalSequence.to_state_sequence` callers
   (tests, docs) to `StateSequence.from_intervals`.
3. Delete `IntervalSequence`, `BaseSequence`, `infer_sequence_type`,
   related `SequenceConfig` fields.
4. Simplify `io/loaders.py` return types and signatures; drop
   `sequence_type` param.
5. Delete `PatternCriterion` + `TestPatternCriterion`.
6. Delete `visualization/` + tests + `demo_showcase.ipynb`.
7. Drop `plotnine`/`matplotlib` from `pyproject.toml`. Run `uv lock` and
   commit the refreshed lockfile in the same PR.
8. Update `CHANGELOG.md`, `README.md` (verify no stale viz examples), and
   `CLAUDE.md` (architecture diagram, module-level guidance — remove the
   `### visualization/` section and `IntervalSequence` references).
9. Bump version in `pyproject.toml` to `0.4.0`.
10. Run full test suite, ruff, mypy. Fix anything that surfaces.
11. Hand off to the `yasqat-release` skill.

Steps 1–2 are additive; the existing test suite passes through them.
Step 3 onwards is destructive — by then, callers are already on the new
API, so the failures from deletion are isolated to the deletion itself.
