_Date: 2026-03-04_

## Summary

Build a Quarto-based documentation website for the yasqat Python library, hosted on GitHub Pages, living in `/docs` at the root of the repository. API reference is auto-generated from docstrings using `quartodoc`. Design is guided by the `design-with-taste` and `design-taste-frontend` skill principles (Family.co philosophy + senior UI/UX engineering standards).

---

## Decisions Made

| Question | Decision | Rationale |
|---|---|---|
| Framework | Quarto website | Best Python/Jupyter interop; native `.ipynb` rendering; `quartodoc` support |
| API reference | Auto-generated via `quartodoc` | ~50 public symbols — must stay in sync with code automatically |
| Location | `/docs` in same repo | Standard for single-maintainer libraries; no separate repo needed |
| Hosting | GitHub Pages | Free, integrated with GitHub Actions |
| Package publish | No change needed | `pyproject.toml` already restricts wheel to `src/yasqat`; `/docs` is naturally excluded |
| Branch | `doc` (off `dev`) | Isolated from release workflow |

## Framework: Quarto + quartodoc

**Quarto** (website format, not book) gives:
- Navbar + per-section sidebar navigation
- Native `.ipynb` rendering (demo notebook becomes a page)
- Bootstrap 5 theming via custom SCSS
- GitHub Actions integration for Pages deploy

**quartodoc** gives:
- Auto-generated `.qmd` API pages from Python docstrings
- One YAML configuration block in `_quarto.yml`
- Regenerates on every build — never goes stale

**Dependencies** (dev-only, not in package):
```
quarto (CLI, installed separately — not pip)
quartodoc (pip install quartodoc)
```

## Design System

### Philosophy

Guided by the Family.co design principles (Simplicity → Fluidity → Delight) and senior UI/UX engineering standards. Primary rules applied:

- **No generic AI aesthetic**: no Inter, no purple/neon, no centered hero, no 3-equal-card grids
- **Single accent color**, saturation < 80%
- **Left-aligned hero** with asymmetric layout (text left, code right)
- **Progressive disclosure** in sidebar (collapsed sections unfold, not all visible at once)
- **Subtle fluidity**: CSS transitions on hover/active states; no JS required for basic feel

### Typography

| Role | Font | Weight/Style | Notes |
|---|---|---|---|
| Display / Headers | **Outfit** (Google Fonts) | 700, `letter-spacing: -0.02em` | Replaces Inter (banned) |
| Body | **Outfit** | 400, `line-height: 1.7`, `max-width: 65ch` | Readable measure |
| Code | **JetBrains Mono** | 400/500 | For all code blocks and inline code |

Import snippet for SCSS:
```scss
@import url('https://fonts.googleapis.com/css2?family=Outfit:wght@400;600;700&family=JetBrains+Mono:wght@400;500&display=swap');
```

### Color Palette

| Role | Name | Hex |
|---|---|---|
| Background | Zinc-50 | `#fafafa` |
| Primary text | Zinc-900 | `#18181b` |
| Muted text | Zinc-500 | `#71717a` |
| **Accent** | **Deep Teal** | **`#0f766e`** |
| Accent hover | Teal-800 | `#115e59` |
| Navbar bg | Zinc-900 | `#18181b` |
| Navbar text | White | `#ffffff` |
| Code block bg | Zinc-100 | `#f4f4f5` |
| Borders | Zinc-200 | `#e4e4e7` |
| Callout left-border | Deep Teal | `#0f766e` |

**Why Deep Teal?** Unusual for Python library docs (which default to blue/purple). Desaturated enough to be elegant. Evokes trajectory and flow — appropriate for a sequence analytics library targeting life-course research.

### Fluidity & Delight (CSS-only, no JS)

Docs sites are high-frequency-use → subtle micro-interactions only (per Delight-Impact Curve):

```scss
// Sidebar active link transition
.sidebar-item a {
  transition: color 0.15s cubic-bezier(0.16, 1, 0.3, 1),
              border-color 0.15s cubic-bezier(0.16, 1, 0.3, 1);
}

// Copy button on code blocks — visible on hover only
.code-copy-button {
  opacity: 0;
  transition: opacity 0.2s cubic-bezier(0.16, 1, 0.3, 1);
}
pre:hover .code-copy-button { opacity: 1; }

// Callout left-border style (teal, no loud background)
.callout { border-left: 3px solid #0f766e; background: transparent; }
.callout-tip { border-left-color: #0f766e; }
.callout-warning { border-left-color: #d97706; }
.callout-note { border-left-color: #71717a; }
```

## GitHub Actions — Docs Workflow

New file: `.github/workflows/docs.yml` (separate from existing `publish.yml`).

Trigger: push to `main` branch (docs deployed when code ships, not on `doc` branch).

```yaml
name: Build and deploy docs
on:
  push:
    branches: [main]
  workflow_dispatch:   # allow manual trigger

jobs:
  deploy:
    runs-on: ubuntu-latest
    permissions:
      contents: write  # needed to push to gh-pages

    steps:
      - uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.11"

      - name: Install Quarto
        uses: quarto-dev/quarto-actions/setup@v2

      - name: Install yasqat and quartodoc
        run: pip install -e ".[dev]" quartodoc

      - name: Generate API docs
        run: quartodoc build --config docs/_quarto.yml
        working-directory: .

      - name: Render docs
        run: quarto render docs/

      - name: Deploy to GitHub Pages
        uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: docs/_site
          publish_branch: gh-pages
```

**GitHub Pages setup** (one-time manual step): Settings → Pages → Source: `gh-pages` branch.

## Package Publish — No Changes Needed

Current `pyproject.toml` already excludes `/docs`:

```toml
[tool.hatch.build.targets.wheel]
packages = ["src/yasqat"]   # wheel: only source code

[tool.hatch.build.targets.sdist]
include = ["src/", "tests/", "README.md", "LICENSE"]  # sdist: no docs
```

The `/docs` directory is never packaged. No changes required.
